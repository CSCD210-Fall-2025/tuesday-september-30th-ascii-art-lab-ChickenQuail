Part 1: Writing and Running Your First Program

Start by creating AsciiArt.java in the root of your repository. This file should define a public class named AsciiArt with a public static void main(String[] args) method.

Why do we compile before running?

Java source files (.java) are human-readable. Before the computer can run your code, the Java compiler (javac) translates it into bytecode (.class files). The Java Virtual Machine (java) then interprets that bytecode. Attempting to run a .java file directly will result in an error because the JVM requires compiled bytecode to execute.

Inside main, write System.out.println statements to print your artwork. Each line of your picture should be printed by a separate call. For example: 

System.out.println("  /\\");
System.out.println(" /  \\");
System.out.println("/____\\");

When your code is ready, compile and run it:

javac AsciiArt.java
java AsciiArt
Need help with escape sequences?

To include characters that have special meaning in Java strings, you must use escape sequences. The backslash (\) introduces an escape sequence and tells the compiler to treat the next character differently. Here are the most common escapes you will use when printing ASCII art:

  • Backslash: write "\\" to produce a single backslash (\) in your output. Because \ starts an escape sequence, you need two backslashes in a row to represent one literal backslash.
  • Double quote: write \" to include a double quote (") inside a string. Without escaping, a double quote would prematurely end the string.
  • New line: \n creates a line break inside a single string literal. You usually won't need this in simple print statements because System.out.println already moves to the next line, but it's handy when building multi-line strings.
  • Tab: \t inserts a horizontal tab. This can be useful for aligning columns of text; each \t moves the cursor to the next tab stop.
  • Single quote: \' allows you to print a single quote (') inside a character or string literal. In double-quoted strings you don't strictly need to escape single quotes, but it's required inside single-quoted character literals.

Make sure each System.out.println statement ends with a semicolon; forgetting one will cause a compile error. See the Common syntax errors section below for examples.

Part 2: Procedural Decomposition with Static Methods

Once your picture prints correctly, look for patterns: do you repeat the same group of lines? You can extract those into a helper method to avoid duplication.

Define a new static method outside main:

public static void drawCone() {
    System.out.println("   /\\");
    System.out.println("  /  \\");
    System.out.println(" /    \\");
}

Then call it from main wherever you need that pattern:

public static void main(String[] args) {
    drawCone();
    System.out.println("|      |");
    System.out.println("+------+");
    drawCone();
}
Why decompose?

Breaking your program into methods makes it easier to read and maintain. If you need to change the cone shape, you edit the method once and all calls update automatically. Methods also help you name logical tasks; drawCone() clearly communicates its purpose without revealing the details inside.

Try creating methods like drawDiamond(), drawX(), and drawRocket() in your own program. See the files in examples/ for reference.

Part 3: Debugging and Error Types

Chapter 1 introduces three types of errors:

  • Syntax errors: Violations of Java grammar, such as missing semicolons, braces, or incorrect capitalization. The compiler catches these.
  • Runtime errors: Errors that occur while the program runs, like dividing by zero or accessing an invalid array index. These cause your program to crash.
  • Logic errors: Mistakes in your algorithm that produce wrong output even though the program compiles and runs.
Practice spotting syntax errors

In extras/LogicErrorExample.java, the program prints 1 + 1 = 11 instead of 2 because of string concatenation. Fixing these will help you understand runtime and logic errors.

Common syntax errors and reading error messages

New programmers often make the same kinds of syntax mistakes. When you compile, the Java compiler tells you what went wrong, but the messages can be cryptic at first. Here are some tips and examples using the files in the errors/ folder:

How to read a compiler error

An error message typically includes the name of the file and the line number where the compiler got confused. The text after error: describes what it was expecting or what it couldn't find. A caret (^) appears on the next line to point underneath the code near the problem. Focus on the first error reported and fix that one; many subsequent errors are just side effects of the first.

Examples to try

Compile each of these files with javac to see the corresponding error messages. Then read the comments in each file and use the hints below to correct them:

  • MissingSemicolon.java - The System.out.println call is missing its ending semicolon. You'll see an error like: 
    MissingSemicolon.java:4: error: ';' expected
    System.out.println("Hello world")
                                    ^
1 error
    The compiler expected a semicolon at the end of the statement. Fix it by adding ; after the closing parenthesis.
  • UnclosedString.java - A string literal is missing its closing double quote. The compiler will report an unclosed string literal and point near the missing quote: 
    UnclosedString.java:4: error: unclosed string literal
    System.out.println("This string never ends);
                                    ^
1 error
    Make sure every opening quote has a matching closing quote and that special characters inside the string are properly escaped.
  • WrongCase.java - Java is case sensitive. Here the code uses system.out instead of System.out. The compiler cannot find a variable named system, so you get: 
    WrongCase.java:4: error: cannot find symbol
    system.out.println("Oops wrong case");
    ^
  symbol:   variable system
  location: class WrongCase
1 error
    Capitalise System correctly to fix the error.
  • MissingBrace.java - When braces don't match, the compiler eventually reaches the end of the file and reports something like reached end of file while parsing: 
    MissingBrace.java:6: error: reached end of file while parsing
}
^
1 error
    Check that every opening brace { has a corresponding closing brace }. In this file the main method ends without its own closing brace.
  • MissingParenthesis.java - If you forget a closing parenthesis, you'll see an error like ')' expected depending on what's missing: 
    MissingParenthesis.java:4: error: ')' expected
    System.out.println("Missing parenthesis"
                                               ^
1 error
    Add the missing ) to match the opening parenthesis in the method call.

As you fix each mistake and recompile, the error messages will disappear. Eventually, the compiler will succeed and you can run the program. Developing the habit of reading and understanding compiler feedback early will save you lots of time later!

Using GitHub Pages

This lab uses the docs/ folder to store the website. Once you push your repository to GitHub Classroom, you can enable GitHub Pages to serve these docs as a website:

  1. Go to your repository on GitHub.
  2. Click Settings > Pages.
  3. Under Source, choose Deploy from a branch and select the main branch and docs/ folder. Click Save.
  4. After a minute, your site will be live at https://<your-username>.github.io/<your-repo-name>/. Replace the placeholders with your GitHub username and repository name.

GitHub Pages will automatically serve any static HTML files in the docs/ directory. You can customize this site further by editing these HTML files or adding new pages.
